交互方式 2：点击公式 → 展开公式卡 + 追问

Formula Click → Expanded Card → Interactive
核心机制
模型的总损失函数由两部分组成，通过 λ 平衡任务学习和检索质量：
$$ \mathcal{L}_{total} = \mathcal{L}_{task} + \lambda \cdot \mathcal{L}_{retrieval} $$
点击查看详解 ↓
公式核心 公式支持
收起 ▲
符号拆解
符号含义鼓励什么惩罚什么去掉会怎样
L_task任务损失正确预测预测偏差模型不学任务
L_retrieval检索损失检索相关文档检索噪声检索质量下降
λ平衡系数控制两部分权重—失衡：太大忽略检索，太小过度依赖
🔢 小数字例子
假设 L_task=0.8, L_retrieval=0.3, λ=0.5
→ L_total = 0.8 + 0.5×0.3 = 0.95
如果去掉 λ·L_retrieval，loss=0.8 看起来更低，但模型会忽略检索质量，回答变得不准确。

交互方式 3：段落悬停 → 快捷追问入口

Paragraph Hover → Quick Ask Button
深挖推导
标准注意力机制的计算复杂度是 O(n²d)，其中 n 是序列长度，d 是维度。这意味着处理 1000 个 token 需要 100 万次注意力计算。
💬 问
为了解决这个问题，作者提出了稀疏注意力：只计算最重要的 token 对之间的注意力，将复杂度降到 O(n√n)。
💬 问
UX 设计： 段落左侧细线 + 悬停变色 + 右侧「问」按钮。点击后自动将段落文本作为 context 发送到追问面板。不需要用户手动复制粘贴。

数据流

用户选中文字 → selectionStore.setSelectedText(text, blockId, evidenceRef)

点击「追问」→ askPanel.prefillWithSelection()

发送追问 → 后端 /api/interactive/ask 接收：
{ question, selected_text, block_id, evidence_ref, card_id, paper_id }

后端构造完整上下文包 → LLM 生成回答 → WebSocket 流式返回

回答中标注证据来源 → 点击可跳转到原文位置
